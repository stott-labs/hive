# Hivemind Configuration

Hivemind uses two local config files, both generated by the setup wizard and gitignored from the repo.

## Config File: `~/.config/hivemind/config.md`

A YAML-style Markdown file containing org-specific values. Every skill reads this at runtime.

### Azure DevOps Example

```yaml
provider: ado

# Identity
name: John Developer
email: john@company.com

# Azure DevOps
ado_org: mycompany
ado_project: My Project
ado_project_id: 00000000-0000-0000-0000-000000000000

# Reviewer IDs (ADO identity GUIDs)
reviewer_ids:
  - 00000000-0000-0000-0000-000000000001
  - 00000000-0000-0000-0000-000000000002

# Repositories (name → ADO repo ID mapping)
repos:
  api:
    name: my-api
    id: 00000000-0000-0000-0000-000000000010
  web:
    name: my-web
    id: 00000000-0000-0000-0000-000000000011

# Paths
projects_dir: /path/to/your/projects
hive_dir: /path/to/your/hive

# Documentation (optional)
docs_dir: /path/to/your/docs
docs_bugs_path: Bugs/
```

### GitHub Example

```yaml
provider: github

name: Jane Developer
email: jane@company.com

github_org: mycompany
github_user: janedev

default_reviewers:
  - reviewer-one
  - reviewer-two

repos:
  - my-api
  - my-web

projects_dir: /path/to/your/projects
hive_dir: /path/to/your/hive

docs_dir: /path/to/your/docs
docs_bugs_path: Bugs/
```

### Key Fields

| Field | Required | Description |
|-------|----------|-------------|
| `provider` | Yes | `ado` or `github` |
| `name` | Yes | Developer display name |
| `email` | Yes | Developer email |
| `ado_org` | ADO only | Azure DevOps organization name |
| `ado_project` | ADO only | ADO project name |
| `ado_project_id` | ADO only | ADO project GUID (for work item linking) |
| `reviewer_ids` | ADO only | List of ADO identity GUIDs for PR reviewers |
| `github_org` | GitHub only | GitHub organization name |
| `github_user` | GitHub only | Your GitHub username |
| `default_reviewers` | GitHub only | GitHub usernames for PR reviewers |
| `repos` | Yes | Repositories to track (format varies by provider) |
| `projects_dir` | Yes | Base directory where repos live |
| `hive_dir` | No | Path to HIVE installation (enables `/dashboard`) |
| `docs_dir` | No | Path to documentation directory |
| `docs_bugs_path` | No | Subdirectory for bug documentation files |

## Paths File: `~/.config/hivemind/paths.env`

A shell-sourceable environment file for runtime path discovery:

```bash
HIVEMIND_DIR="/path/to/hivemind"
PROJECTS_DIR="/path/to/projects"
HIVE_DIR="/path/to/hive"
```

Skills source this file to resolve directories without re-parsing the config. It's particularly important on Windows where path resolution can be tricky.

## Example Config Template

The repo includes `hivemind.config.example.md` — a committed, annotated template showing all possible fields. It serves as reference documentation and a starting point for manual configuration.

## Configuration Precedence

Skills look for config in this order:

1. `~/.config/hivemind/config.md` (preferred — generated by setup)
2. `{hivemind_repo}/hivemind.config.md` (fallback — for development/testing)

## Setup Behavior

### First-time setup (no existing config)

The wizard prompts for everything: identity, provider, provider-specific fields, paths, and documentation directories.

### Setup with existing HIVE config

If HIVE was already set up and `~/.config/hivemind/config.md` exists, the wizard:
- Skips identity and provider questions
- Only installs skills and updates paths
- Creates platform CLAUDE.md symlink

### Re-running setup

Setup is idempotent. Run it again to update any config values. Existing symlinks are replaced, not duplicated.

## Environment Variables

These are set in your shell or HIVE's `.env`, not in the Hivemind config:

| Variable | Required For | Purpose |
|----------|-------------|---------|
| `ADO_PAT` | ADO provider | Azure DevOps Personal Access Token |
| `GITHUB_TOKEN` | GitHub provider | GitHub Personal Access Token |
